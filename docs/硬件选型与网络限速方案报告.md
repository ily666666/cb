## 二、行业参考案例：智慧矿山视频违章检测的云边协同实践

### 2.1 案例背景

在已落地的智慧矿山视频违章检测项目中，"云端+边缘"协同的 AI 智能检测方案已得到充分验证。该项目覆盖综采工作面、主运皮带、掘进巷道、架空乘人装置等高风险区域，部署多台矿用防爆监控摄像头，实现井上井下核心场景无死角覆盖。

核心检测功能包括：人员违章行为识别（未佩戴安全帽、违规闯入危险区域、携带违禁物品、摔倒求救等）、设备异常状态监测（皮带跑偏、堆煤、滚筒打滑、设备泄漏等）。系统试运行期间有效过滤无效预警，成功预警多起皮带跑偏、异物入侵隐患。

### 2.2 案例硬件部署

该矿山项目采用昆仑芯系列加速卡，构建如下云边架构：

| 部署位置 | 硬件配置 | 单卡算力（FP16 / INT8） | 总算力（FP16） | 总算力（INT8） | 核心任务 |
|---------|---------|----------------------|--------------|--------------|---------|
| 云端 | 1台P800设备（内置8张P800卡） | 345 TFLOPS / 690 TOPS | 2760 TFLOPS | 5520 TOPS | 大模型二次复检、模型训练、全局数据管理 |
| 边缘 | 2台R200设备（每台4张R200卡） | 128 TFLOPS / 256 TOPS | 1024 TFLOPS | 2048 TOPS | 视频帧实时筛查、违章快速预警 |

**云端/边缘总算力比 = 2760 / 1024 ≈ 2.7 倍。**

### 2.3 案例验证的核心设计原则

该矿山项目的成功运行验证了以下云边协同设计原则，也是本项目 GPU 选型的核心参照依据：

1. **边缘端承载全量数据的实时推理**：边缘端运行轻量化算法处理 100% 的视频流，优先保障检测速度与多路并行处理能力。
2. **云端仅处理边缘筛出的少量疑似数据**：边缘初筛后，仅约 5% 的疑似违章帧回传云端进行大模型二次复核，云端数据量大幅收窄。
3. **云端单卡性能高于边缘**：云端以更强的单卡算力运行更大更精准的复核模型，同时承担模型训练与迭代任务。
4. **算力按需配比，非简单堆叠**：云端算力规模匹配其实际处理的数据量，边缘算力规模匹配全量实时处理需求。

## 三、本项目显卡选型方案

### 3.1 选型方案：RTX 4080 + RTX 4070

| 层级 | 显卡型号 | FP32 算力 | FP16 Tensor 算力 | INT8 Tensor 算力 | 显存 | 参考价 | 数量 | 小计 |
|------|---------|----------|-----------------|-----------------|------|--------|------|------|
| 云 (1台) | **RTX 4080 16GB** | 48.7 TFLOPS | 97.5 TFLOPS | 195.0 TOPS | 16 GB | ~¥7,500 | 1 | ¥7,500 |
| 边 (2台) | **RTX 4070 12GB** | 29.2 TFLOPS | 58.3 TFLOPS | 116.6 TOPS | 12 GB | ~¥3,800 | 2 | ¥7,600 |
| 端 (4台) | 无 GPU（CPU 即可） | - | - | - | - | - | 4 | ¥0 |
| | | | | | | **GPU 总计** | | **¥15,100** |

两款卡均基于 NVIDIA Ada Lovelace 架构、第4代 Tensor Core，核心规格对比：

| 参数 | RTX 4080 | RTX 4070 | 比值 |
|------|---------|---------|------|
| Tensor Core | 304 个 | 184 个 | 1.65× |
| Boost 频率 | 2505 MHz | 2475 MHz | 1.01× |
| FP16 Tensor (dense) | 97.5 TFLOPS | 58.3 TFLOPS | **1.67×** |
| INT8 Tensor (dense) | 195.0 TOPS | 116.6 TOPS | **1.67×** |
| 显存 | 16 GB GDDR6X | 12 GB GDDR6X | 1.33× |
| 显存带宽 | 717 GB/s | 504 GB/s | 1.42× |

### 3.2 算力比分析

**单卡倍速（RTX 4080 ÷ RTX 4070）：**

- FP16：97.5 ÷ 58.3 ≈ **1.67 倍**
- INT8：195.0 ÷ 116.6 ≈ **1.67 倍**
- FP32：48.7 ÷ 29.2 ≈ **1.67 倍**

同架构下各精度比值完全一致，RTX 4080 单卡性能为 RTX 4070 的 **1.67 倍**。

**部署总算力（云端 ÷ 边缘）：**

| 精度 | 云端（1×4080） | 边缘（2×4070） | 云端/边缘 |
|------|--------------|--------------|----------|
| FP16 | 97.5 TFLOPS | 116.6 TFLOPS | 0.84 |
| INT8 | 195.0 TOPS | 233.2 TOPS | 0.84 |

### 3.3 选型合理性论证——与矿山案例的对标分析

矿山案例中云端总算力是边缘的 2.7 倍，而本项目云端总算力仅为边缘的 0.84 倍。两者比值差异显著，但选型逻辑完全一致——**算力配比取决于各层级实际处理的数据量和模型复杂度**，而非机械地追求固定比值。以下从四个维度对标论证：

#### （1）云端数据量远小于矿山案例，单卡 1.67× 优势已充分覆盖

矿山案例中，边缘端筛出约 **5%** 的疑似违章帧回传云端复核。本项目实测数据显示，边侧置信度筛选后仅约 **0.25%** 的低置信度样本转发云端——**云端实际处理的数据量仅为矿山案例的 1/20**。

RTX 4080 单卡算力为 RTX 4070 的 1.67 倍，处理这 0.25% 的数据绑绑有余。矿山案例之所以堆叠 2.7 倍云端算力，是因为视频帧的大模型复核（如多模态融合、大参数视觉模型）计算量远高于本项目的 1D 信号分类推理。

#### （2）边缘端承载全量数据实时推理，与矿山案例架构一致

两个项目的边缘端定位完全一致：**处理 100% 的输入数据，运行轻量化模型实时推理**。

矿山案例用 2 台 R200 设备（共 8 张卡，总算力 1024 TFLOPS FP16）承载多路高清视频流实时解析。本项目用 2× RTX 4070（总算力 116.6 TFLOPS FP16）承载雷达/通信信号流实时分类。算力量级差异反映的是任务本身的计算密度差异（视频帧 vs 1D 信号），而非架构设计的差异。

本项目边侧模型 real_resnet20 仅 4.5M 参数（18MB 权重），推理显存 ~200MB，单张 RTX 4070（12GB 显存、58.3 TFLOPS）即可高效运行；双卡部署提供并行吞吐与冗余备份，与矿山案例的多设备互备策略一致。

#### （3）云端大模型推理 + 训练需求，RTX 4080 16GB 显存充分覆盖

本项目云侧模型 complex_resnet50 + 注意力机制，参数量 ~94M（378MB 权重）：

| 云侧任务 | 显存需求 | RTX 4080 16GB 余量 |
|---------|---------|-------------------|
| 推理 (batch=128) | ~1.5 GB | 14.5 GB 空闲 |
| 预训练 (batch=32，含梯度+优化器) | ~4 GB | 12 GB 空闲 |
| 知识蒸馏 (教师+学生同时加载) | ~5 GB | 11 GB 空闲 |

16GB 显存相比实际需求有 **3~10 倍** 余量，完全满足云端大模型推理、预训练教师模型、联邦聚合等全部任务，与矿山案例中云端"大显存 + 高单卡性能"的配置理念一致。

同时，RTX 4080 的 717 GB/s 显存带宽较 RTX 4070 高出 42%，在模型训练（频繁读写梯度和参数）场景下优势更加明显。

#### （4）经济性优势：按需配比 vs 过度堆叠

| 方案 | GPU 成本 | 云端利用率 |
|------|---------|----------|
| 原方案 RTX 4090 + 2×4070 Ti | ~¥22,600 | 云端 24GB 显存利用率 < 25% |
| **本方案 RTX 4080 + 2×4070** | **~¥15,100** | 云端 16GB 显存利用率 25%~31% |

节省约 **¥7,500（33%）**。矿山案例部署数百万算力的昆仑芯集群，是因为多路高清视频实时处理 + 大参数视觉模型的刚性需求。本项目云侧模型仅 94M 参数，推理显存 1.5GB，选择 RTX 4090 存在明显的算力和显存冗余。RTX 4080 以 1.67 倍的单卡性能优势 + 16GB 显存，精准匹配云端"少量数据、高精度复核"的定位，是更具性价比的选择。

### 3.4 与矿山案例的架构对齐汇总

| 设计原则 | 矿山案例（昆仑芯） | 本项目（NVIDIA RTX 40系） |
|---------|-------------------|------------------------|
| 整体架构 | 云端精准复核 + 边缘实时筛查 | 云端精准复核 + 边缘实时筛查 |
| 边缘端定位 | 处理 100% 视频流，轻量化算法实时推理 | 处理 100% 信号流，轻量模型实时推理 |
| 云端定位 | 处理约 5% 疑似帧，大模型二次复核 + 训练 | 处理约 0.25% 低置信度样本，大模型复核 + 训练 |
| 边缘算力配置 | 2台×4卡 R200（多卡并行+冗余） | 2× RTX 4070（双卡并行+冗余） |
| 云端算力配置 | 1台×8卡 P800（高单卡性能+大显存） | 1× RTX 4080（1.67倍单卡性能+16GB显存） |
| 云端单卡/边缘单卡 | P800 345 TFLOPS / R200 128 TFLOPS = **2.7×** | 4080 97.5 TFLOPS / 4070 58.3 TFLOPS = **1.67×** |
| 云端数据处理占比 | ~5% | ~0.25% |
| 算力配比逻辑 | 视频大模型复核计算量大 → 云端总算力 2.7× 边缘 | 1D 信号分类复核计算量小 → 云端单卡 1.67× 即满足 |

### 3.5 选型结论

本项目 GPU 选型遵循经过智慧矿山视频违章检测案例验证的 **"边缘重吞吐、云端重精度"** 云边协同设计原则：

- **边缘端** 以 2× RTX 4070 提供充足的并行推理吞吐与冗余保障，承载全量信号流实时分类与置信度筛选；
- **云端** 以 1× RTX 4080 凭借 **1.67 倍** 的单卡性能优势和 16GB 大显存，聚焦约 0.25% 低置信度样本的精准复核，同时承担教师模型预训练、知识蒸馏、联邦聚合等训练任务；
- 相较于矿山案例处理多路高清视频流的重算力需求，本项目面向 1D 信号分类任务，模型参数量和推理计算量均低一个量级，无需在云端过度堆叠算力；
- RTX 4080 + 2× RTX 4070 的方案以 **¥15,100** 的 GPU 成本，在充分保障检测精度与实时性的前提下实现经济性最优。

## 四、网络限速方案

### 4.1 固定带宽设定

| 链路 | 通信方式 | 限速值 | 换算 |
|------|---------|--------|------|
| 端↔边 | 无人船/无人机无线自组网 | **100 MB/s** | 800 Mbps |
| 边↔云 / 端↔云 | 散射通信 | **8 MB/s** | 64 Mbps |

### 4.2 方案一：企业可网管交换机端口限速

最简单的方案——**在交换机上按端口设速率上限**，零侵入，不需要在任何机器上装软件或执行命令。

#### 4.2.1 物理拓扑

```
                  企业可网管交换机（千兆）
         ┌─────────────────────────────────┐
         │  Port 1  ←  限速 64Mbps (8MB/s)  │ ← 云节点
         │  Port 2  ←  限速 800Mbps(100MB/s)│ ← 边1
         │  Port 3  ←  限速 800Mbps(100MB/s)│ ← 边2
         │  Port 4  ←  不限速               │ ← 端1
         │  Port 5  ←  不限速               │ ← 端2
         │  Port 6  ←  不限速               │ ← 端3
         │  Port 7  ←  不限速               │ ← 端4
         └─────────────────────────────────┘
```

#### 4.2.2 生效原理

数据经过两个端口，**以慢的那个为准**：

| 通信方向 | 发送端口 | 接收端口 | 实际速率 = min(发,收) |
|---------|---------|---------|---------------------|
| 端→边 | 不限 | 800 Mbps | **100 MB/s** |
| 边→端 | 800 Mbps | 不限 | **100 MB/s** |
| 边→云 | 800 Mbps | 64 Mbps | **8 MB/s** |
| 云→边 | 64 Mbps | 800 Mbps | **8 MB/s** |
| 端→云 | 不限 | 64 Mbps | **8 MB/s** |
| 云→端 | 64 Mbps | 不限 | **8 MB/s** |

**凡是跟云通信 → 8 MB/s；端和边之间通信 → 100 MB/s。**

#### 4.2.3 推荐交换机

| 型号 | 端口 | 参考价 | 说明 |
|------|------|--------|------|
| **TP-Link TL-SG2008** | 8口千兆 | ~¥200 | 支持端口出入双向限速，Web 管理 |
| H3C S1850V2-10P | 10口千兆 | ~¥350 | 企业级，功能更全 |
| 华为 S1720-8GW-2P | 8口千兆 | ~¥400 | 华为品牌 |

#### 4.2.4 交换机配置步骤（以 TP-Link TL-SG2008 为例）

1. 网线连接电脑和交换机，浏览器访问 `192.168.0.1`
2. 登录管理页面
3. 进入 **QoS → 速率限制（Rate Limit）**
4. 设置端口限速：

| 端口 | 入口限速 | 出口限速 | 连接设备 |
|------|---------|---------|---------|
| Port 1 | 64 Mbps | 64 Mbps | 云节点 |
| Port 2 | 800 Mbps | 800 Mbps | 边节点 1 |
| Port 3 | 800 Mbps | 800 Mbps | 边节点 2 |
| Port 4-7 | 不限 | 不限 | 端节点 1-4 |

5. 保存/应用

**完成。不需要在任何主机上做配置。**

#### 4.2.5 优缺点

| 优点 | 缺点 |
|------|------|
| 配置简单，一处搞定 | 不能模拟延迟和丢包 |
| 零侵入，不动主机 | 需要买可网管交换机（~¥200） |
| 更真实，瓶颈在网络设备上 | 修改限速值需登录交换机 |

### 4.3 方案二：Linux `tc` 软件限速

不依赖特殊交换机，在每台机器上用 Linux `tc` 命令做内核级限速。用普通交换机或路由器互联即可。

#### 4.3.1 物理拓扑

```
                  普通交换机/路由器（千兆）
         ┌──────────────────────────────┐
         │  所有端口不限速               │
         └─┬────┬────┬────┬───┬───┬───┬─┘
           │    │    │    │   │   │   │
        ┌──┴┐┌─┴──┐┌┴──┐┌┴─┐┌┴─┐┌┴─┐┌┴─┐
        │云  ││边1 ││边2 ││端1││端2││端3││端4│
        │tc  ││tc  ││tc  ││tc ││tc ││tc ││tc │
        └───┘└────┘└────┘└──┘└──┘└──┘└──┘

     限速靠每台机器上的 tc 命令，按目标 IP 区分链路
```

#### 4.3.2 原理

```
  应用程序发出数据包
        ↓
  ┌───────────────────────────────────┐
  │         tc (内核流量控制)           │
  │                                   │
  │  u32 filter: 判断目标 IP          │
  │    ├→ 目标是边侧 IP → 800Mbps     │
  │    └→ 目标是云侧 IP → 64Mbps      │
  └───────────────────────────────────┘
        ↓
     网卡 eth0 发出
```

#### 4.3.3 各节点配置命令

**IP 分配：**

```
云:  192.168.1.10
边1: 192.168.1.21    边2: 192.168.1.22
端1: 192.168.1.31    端2: 192.168.1.32    端3: 192.168.1.33    端4: 192.168.1.34
```

**端节点（端→边 100MB/s，端→云 8MB/s）：**

```bash
IFACE=eth0

sudo tc qdisc del dev $IFACE root 2>/dev/null
sudo tc qdisc add dev $IFACE root handle 1: htb default 99
sudo tc class add dev $IFACE parent 1: classid 1:99 htb rate 1000mbit

# 端→边：800Mbps (100 MB/s)
sudo tc class add dev $IFACE parent 1: classid 1:10 htb rate 800mbit ceil 800mbit
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.21/32 flowid 1:10
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.22/32 flowid 1:10

# 端→云：64Mbps (8 MB/s)
sudo tc class add dev $IFACE parent 1: classid 1:20 htb rate 64mbit ceil 64mbit
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.10/32 flowid 1:20
```

**边节点（边→云 8MB/s，边→端 100MB/s）：**

```bash
IFACE=eth0

sudo tc qdisc del dev $IFACE root 2>/dev/null
sudo tc qdisc add dev $IFACE root handle 1: htb default 99
sudo tc class add dev $IFACE parent 1: classid 1:99 htb rate 1000mbit

# 边→云：64Mbps (8 MB/s)
sudo tc class add dev $IFACE parent 1: classid 1:10 htb rate 64mbit ceil 64mbit
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.10/32 flowid 1:10

# 边→端：800Mbps (100 MB/s)
sudo tc class add dev $IFACE parent 1: classid 1:20 htb rate 800mbit ceil 800mbit
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.31/32 flowid 1:20
```

**云节点（云→边 8MB/s）：**

```bash
IFACE=eth0

sudo tc qdisc del dev $IFACE root 2>/dev/null
sudo tc qdisc add dev $IFACE root handle 1: htb default 99
sudo tc class add dev $IFACE parent 1: classid 1:99 htb rate 1000mbit

# 云→边：64Mbps (8 MB/s)
sudo tc class add dev $IFACE parent 1: classid 1:10 htb rate 64mbit ceil 64mbit

  match ip dst 192.168.1.21/32 flowid 1:10
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.22/32 flowid 1:10
```

**清除所有限速（任意节点）：**

```bash
sudo tc qdisc del dev eth0 root
```

#### 4.3.5 优缺点

| 优点 | 缺点 |
|------|------|
| 免费，无需额外硬件 | 需要在每台机器上配置 |
| 支持模拟延迟和丢包（netem） | 需要 root 权限 |
| 灵活，可随时调整限速值 | 重启后需重新设置（可加开机自启） |

### 4.4 两种方案对比

| 维度 | 交换机端口限速 | Linux `tc` |
|------|--------------|-----------|
| 配置复杂度 | **低**，Web 页面点几下 | 中，每台机器执行脚本 |
| 是否改主机 | **不改，零侵入** | 需要 root 权限 |
| 额外成本 | 交换机 ~¥200 | 免费 |
| 带宽限制 | 精确 | 精确 |
| 延迟/丢包模拟 | **不支持** | **支持**（netem） |
| 真实度 | **更真实** | 较真实 |
| 适用场景 | 正式实验、演示 | 开发调试、需模拟延迟时 |

> 两种方案可组合使用：交换机管带宽上限，tc 叠加延迟/丢包。**但不要在两处同时限带宽，否则会叠加。**

### 4.5 验证限速效果

```bash
# 1. iperf3 测带宽
iperf3 -s                          # 目标节点启动服务端
iperf3 -c <目标IP> -t 10           # 测试节点发起测试

# 预期结果：
#   边→云: ~64 Mbps  (8 MB/s)
#   端→边: ~800 Mbps (100 MB/s)

# 2. 实际文件传输验证
scp testfile.bin user@<云IP>:/tmp/  # 观察速率应 ~8 MB/s
scp testfile.bin user@<边IP>:/tmp/  # 观察速率应 ~100 MB/s
```

### 4.6 项目代码适配

使用物理限速（交换机或 tc）后，项目代码中的 `simulate_bandwidth_mbps` 应设为 `null`，让真实网络延迟自然生效：

```json
{
    "simulate_bandwidth_mbps": null
}
```

> **切勿同时使用物理限速和代码中的 simulate_transfer()，否则延迟会叠加。**

## 五、10GB 数据推理评估

### 4.1 数据规模

| 数据集 | 单样本大小 | 10GB 对应样本数 |
|--------|----------|---------------|
| Link11 (1024点, complex64) | 8 KB | **~125 万条** |
| RML2016 (600点, complex64) | 4.8 KB | **~208 万条** |
| Radar (500点, complex64) | 4 KB | **~250 万条** |

### 4.2 硬件能力评估

| 资源 | 需求 | RTX 4080 / 4070 是否满足 |
|------|------|---------------------------|
| GPU 显存 | 分批推理，每批 ~1MB | 完全满足（batch=128 逐批处理） |
| 系统内存 | 加载 10GB 数据 + 开销 ≈ 20-25GB | 需要 **32GB RAM** |
| 存储 | 10GB PKL 文件 | 需 SSD，HDD 读取会很慢 |

### 4.3 各环节耗时预估（Link11 10GB，基于实测数据线性缩放）

> 实测数据（5000样本）：纯云推理138.15s，协同推理6.11s（边5.77s+云0.34s），推理加速比22.6倍，卸载率0.25%。缩放系数k≈18.7。

**协同计算 T1：**

| 环节 | 计算方式 | 耗时 |
|------|---------|------|
| **端→边传输** | 10GB ÷ 100MB/s | **~102s** |
| **边侧推理** (2×RTX 4070) | 5.77s × 18.7 | **~108s** |
| **边→云传输** (0.25%低置信度 ≈ 25.6MB) | 25.6MB ÷ 8MB/s | **~3.2s** |
| **云侧推理** (RTX 4080, 0.25%样本) | 0.34s × 18.7 | **~6.4s** |
| **资源调度** (北邮) | 毫秒级 | **≤0.005s** |
| **T1 合计** | | **~219s ≈ 3.7 分钟** |

**纯云端计算 T2：**

| 环节 | 计算方式 | 耗时 |
|------|---------|------|
| **端→云传输** | 10GB ÷ 8MB/s | **~1,280s** |
| **云侧推理** (RTX 4080) | 138.15s × 18.7 | **~2,583s** |
| **T2 合计** | | **~3,863s ≈ 64 分钟** |

**T2/T1 = 3,863 / 219 ≈ 17.6 ✓✓✓，远超10倍要求。**

### 4.5 注意事项

1. **内存**：当前代码 `pickle.load()` 一次性加载全量数据到 RAM，10GB 数据需要 **32GB 内存**
2. **存储**：10GB PKL 文件建议放 SSD，HDD 读取 10GB 约需 1-2 分钟额外开销
3. **分批加载**：若内存不足，可改代码为分块加载（需修改 `io_helper.py`），需要时可实现

## 五、评估指标体系

### 5.1 指标公式

```
T_total = T_transfer + T_inference + T_scheduling

其中：
  T_transfer   = T_端→边 + T_边→云            （网络传输时间）
  T_inference  = T_边侧推理 + T_云侧推理        （计算推理时间）
  T_scheduling = T_任务分派 + T_资源分配         （资源调度时间，北邮负责，毫秒级）
```

### 5.2 关键结论

- 10GB 数据协同推理约 **3.7 分钟**，端→云直传约 **64 分钟**，**T2/T1 ≈ 17.6，远超10倍**
- 实测推理加速比达 22.6 倍，云端卸载率仅 0.25%，推理优势是达标的核心
- 协同模式中端→边传输（47%）和边侧推理（49%）占主导，散射链路传输占比极小（1%）
- 北邮毫秒级调度开销占比 < 0.01%
