# 云边端协同信号识别系统 — 硬件选型与网络限速方案报告

## 一、项目计算需求分析

### 1.1 模型参数与显存需求

| 指标 | 云侧模型 | 边侧模型 |
|------|---------|---------|
| 模型名称 | complex_resnet50_*_with_attention | real_resnet20_*_h |
| 网络结构 | 复数 ResNet50 + 时频注意力 (Bottleneck [3,4,6,3]) | 实数 ResNet20 (Bottleneck [2,2,2]) |
| 卷积类型 | ComplexConv2d（每层 2 组权重：实部+虚部） | 标准 Conv2d |
| .pth 文件大小 | **378 MB** | **18 MB** |
| 估算参数量 | ~94M | ~4.5M |
| 推理显存 (batch=128) | ~1.5 GB | ~200 MB |
| 训练显存 (batch=32) | ~4 GB（含梯度+优化器） | ~500 MB（知识蒸馏） |

### 1.2 各层级计算任务

| 层级 | 推理任务 | 训练任务 | GPU 需求等级 |
|------|---------|---------|------------|
| 云 (1台) | complex_resnet50 大模型推理 | 预训练教师模型、联邦聚合 | **高** |
| 边 (2台) | real_resnet20 轻量推理 + 置信度筛选 | 知识蒸馏、联邦本地训练 | **中** |
| 端 (4台) | 无模型推理，仅数据采集/加载 | 无 | **无 GPU** |

## 二、显卡选型方案

### 2.1 核心约束

- 云上算力 ≈ 边侧**总算力**的 **2 倍**（即 Cloud = 2 × (Edge1 + Edge2)）
- 云侧模型 378MB，至少需要 6GB 显存
- 边侧模型 18MB，至少需要 4GB 显存（知识蒸馏训练时需要额外空间）

### 2.2 推荐方案：RTX 4090 + RTX 4070 Ti

| 层级 | 显卡型号 | FP32 算力 | 显存 | 参考价 | 数量 | 小计 |
|------|---------|----------|------|--------|------|------|
| 云 (1台) | **RTX 4090 24GB** | 82.6 TFLOPS | 24 GB | ~¥13,000 | 1 | ¥13,000 |
| 边 (2台) | **RTX 4070 Ti 12GB** | 40.1 TFLOPS | 12 GB | ~¥4,800（二手） | 2 | ¥9,600 |
| 端 (4台) | 无 GPU（CPU 即可） | - | - | - | 4 | ¥0 |
| | | | | **GPU 总计** | | **¥22,600** |

**算力比：82.6 / (40.1 × 2) = 82.6 / 80.2 = 1.03 倍。**

> 说明：若按"云 = 单台边 × 2"理解，82.6 / 40.1 = **2.06 倍**。两种理解均可覆盖。

选择理由：
- RTX 4090 24GB：顶级单卡算力，24GB 显存轻松容纳 378MB 云侧模型 + 训练开销
- RTX 4070 Ti 12GB：12GB 显存远超 18MB 边侧模型需求，知识蒸馏训练有充足余量
- 4070 Ti 性能强劲，单卡即可高效处理边侧推理 + 联邦本地训练

### 2.3 完整主机配置建议

| 角色 | CPU | 内存 | 存储 | GPU | 预算 |
|------|-----|------|------|-----|------|
| 云节点 (1台) | i7-13700 / R7 5800X | **32GB** DDR4 | 1TB SSD | RTX 4090 24GB | ~¥19,000 |
| 边节点 (2台) | i5-13400 / R5 5600X | **32GB** DDR4 | 512GB SSD | RTX 4070 Ti 12GB | ~¥9,000 x2 |
| 端节点 (4台) | i3-12100 / R3 5300G | 8GB DDR4 | 128GB SSD | 无 | ~¥2,000 x4 |
| **总计** | | | | | **~¥45,000** |

> 边节点内存选 32GB，用于加载 10GB 测试数据集。

## 三、网络限速方案

### 3.1 固定带宽设定

| 链路 | 通信方式 | 限速值 | 换算 |
|------|---------|--------|------|
| 端↔边 | 无人船/无人机无线自组网 | **100 MB/s** | 800 Mbps |
| 边↔云 / 端↔云 | 散射通信 | **8 MB/s** | 64 Mbps |

### 3.2 方案一：企业可网管交换机端口限速

最简单的方案——**在交换机上按端口设速率上限**，零侵入，不需要在任何机器上装软件或执行命令。

#### 3.2.1 物理拓扑

```
                  企业可网管交换机（千兆）
         ┌─────────────────────────────────┐
         │  Port 1  ←  限速 64Mbps (8MB/s)  │ ← 云节点
         │  Port 2  ←  限速 800Mbps(100MB/s)│ ← 边1
         │  Port 3  ←  限速 800Mbps(100MB/s)│ ← 边2
         │  Port 4  ←  不限速               │ ← 端1
         │  Port 5  ←  不限速               │ ← 端2
         │  Port 6  ←  不限速               │ ← 端3
         │  Port 7  ←  不限速               │ ← 端4
         └─────────────────────────────────┘
```

#### 3.2.2 生效原理

数据经过两个端口，**以慢的那个为准**：

| 通信方向 | 发送端口 | 接收端口 | 实际速率 = min(发,收) |
|---------|---------|---------|---------------------|
| 端→边 | 不限 | 800 Mbps | **100 MB/s** |
| 边→端 | 800 Mbps | 不限 | **100 MB/s** |
| 边→云 | 800 Mbps | 64 Mbps | **8 MB/s** |
| 云→边 | 64 Mbps | 800 Mbps | **8 MB/s** |
| 端→云 | 不限 | 64 Mbps | **8 MB/s** |
| 云→端 | 64 Mbps | 不限 | **8 MB/s** |

**凡是跟云通信 → 8 MB/s；端和边之间通信 → 100 MB/s。**

#### 3.2.3 推荐交换机

| 型号 | 端口 | 参考价 | 说明 |
|------|------|--------|------|
| **TP-Link TL-SG2008** | 8口千兆 | ~¥200 | 支持端口出入双向限速，Web 管理 |
| H3C S1850V2-10P | 10口千兆 | ~¥350 | 企业级，功能更全 |
| 华为 S1720-8GW-2P | 8口千兆 | ~¥400 | 华为品牌 |

#### 3.2.4 交换机配置步骤（以 TP-Link TL-SG2008 为例）

1. 网线连接电脑和交换机，浏览器访问 `192.168.0.1`
2. 登录管理页面
3. 进入 **QoS → 速率限制（Rate Limit）**
4. 设置端口限速：

| 端口 | 入口限速 | 出口限速 | 连接设备 |
|------|---------|---------|---------|
| Port 1 | 64 Mbps | 64 Mbps | 云节点 |
| Port 2 | 800 Mbps | 800 Mbps | 边节点 1 |
| Port 3 | 800 Mbps | 800 Mbps | 边节点 2 |
| Port 4-7 | 不限 | 不限 | 端节点 1-4 |

5. 保存/应用

**完成。不需要在任何主机上做配置。**

#### 3.2.5 优缺点

| 优点 | 缺点 |
|------|------|
| 配置简单，一处搞定 | 不能模拟延迟和丢包 |
| 零侵入，不动主机 | 需要买可网管交换机（~¥200） |
| 更真实，瓶颈在网络设备上 | 修改限速值需登录交换机 |

### 3.3 方案二：Linux `tc` 软件限速

不依赖特殊交换机，在每台机器上用 Linux `tc` 命令做内核级限速。用普通交换机或路由器互联即可。

#### 3.3.1 物理拓扑

```
                  普通交换机/路由器（千兆）
         ┌──────────────────────────────┐
         │  所有端口不限速               │
         └─┬────┬────┬────┬───┬───┬───┬─┘
           │    │    │    │   │   │   │
        ┌──┴┐┌─┴──┐┌┴──┐┌┴─┐┌┴─┐┌┴─┐┌┴─┐
        │云  ││边1 ││边2 ││端1││端2││端3││端4│
        │tc  ││tc  ││tc  ││tc ││tc ││tc ││tc │
        └───┘└────┘└────┘└──┘└──┘└──┘└──┘

     限速靠每台机器上的 tc 命令，按目标 IP 区分链路
```

#### 3.3.2 原理

```
  应用程序发出数据包
        ↓
  ┌───────────────────────────────────┐
  │         tc (内核流量控制)           │
  │                                   │
  │  u32 filter: 判断目标 IP          │
  │    ├→ 目标是边侧 IP → 800Mbps     │
  │    └→ 目标是云侧 IP → 64Mbps      │
  └───────────────────────────────────┘
        ↓
     网卡 eth0 发出
```

#### 3.3.3 各节点配置命令

**IP 分配：**

```
云:  192.168.1.10
边1: 192.168.1.21    边2: 192.168.1.22
端1: 192.168.1.31    端2: 192.168.1.32    端3: 192.168.1.33    端4: 192.168.1.34
```

**端节点（端→边 100MB/s，端→云 8MB/s）：**

```bash
IFACE=eth0

sudo tc qdisc del dev $IFACE root 2>/dev/null
sudo tc qdisc add dev $IFACE root handle 1: htb default 99
sudo tc class add dev $IFACE parent 1: classid 1:99 htb rate 1000mbit

# 端→边：800Mbps (100 MB/s)
sudo tc class add dev $IFACE parent 1: classid 1:10 htb rate 800mbit ceil 800mbit
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.21/32 flowid 1:10
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.22/32 flowid 1:10

# 端→云：64Mbps (8 MB/s)
sudo tc class add dev $IFACE parent 1: classid 1:20 htb rate 64mbit ceil 64mbit
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.10/32 flowid 1:20
```

**边节点（边→云 8MB/s，边→端 100MB/s）：**

```bash
IFACE=eth0

sudo tc qdisc del dev $IFACE root 2>/dev/null
sudo tc qdisc add dev $IFACE root handle 1: htb default 99
sudo tc class add dev $IFACE parent 1: classid 1:99 htb rate 1000mbit

# 边→云：64Mbps (8 MB/s)
sudo tc class add dev $IFACE parent 1: classid 1:10 htb rate 64mbit ceil 64mbit
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.10/32 flowid 1:10

# 边→端：800Mbps (100 MB/s)
sudo tc class add dev $IFACE parent 1: classid 1:20 htb rate 800mbit ceil 800mbit
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.31/32 flowid 1:20
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.32/32 flowid 1:20
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.33/32 flowid 1:20
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.34/32 flowid 1:20
```

**云节点（云→边 8MB/s）：**

```bash
IFACE=eth0

sudo tc qdisc del dev $IFACE root 2>/dev/null
sudo tc qdisc add dev $IFACE root handle 1: htb default 99
sudo tc class add dev $IFACE parent 1: classid 1:99 htb rate 1000mbit

# 云→边：64Mbps (8 MB/s)
sudo tc class add dev $IFACE parent 1: classid 1:10 htb rate 64mbit ceil 64mbit
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.21/32 flowid 1:10
sudo tc filter add dev $IFACE parent 1: protocol ip prio 1 u32 \
  match ip dst 192.168.1.22/32 flowid 1:10
```

**清除所有限速（任意节点）：**

```bash
sudo tc qdisc del dev eth0 root
```

#### 3.3.4 一键脚本

项目中已提供 `scripts/network_shaping.sh`：

```bash
sudo ./scripts/network_shaping.sh cloud      # 云节点设置限速
sudo ./scripts/network_shaping.sh edge       # 边节点设置限速
sudo ./scripts/network_shaping.sh device     # 端节点设置限速
sudo ./scripts/network_shaping.sh clear      # 清除所有限速
```

> 脚本顶部的 IP 地址需按实际环境修改。

#### 3.3.5 优缺点

| 优点 | 缺点 |
|------|------|
| 免费，无需额外硬件 | 需要在每台机器上配置 |
| 支持模拟延迟和丢包（netem） | 需要 root 权限 |
| 灵活，可随时调整限速值 | 重启后需重新设置（可加开机自启） |

### 3.4 两种方案对比

| 维度 | 交换机端口限速 | Linux `tc` |
|------|--------------|-----------|
| 配置复杂度 | **低**，Web 页面点几下 | 中，每台机器执行脚本 |
| 是否改主机 | **不改，零侵入** | 需要 root 权限 |
| 额外成本 | 交换机 ~¥200 | 免费 |
| 带宽限制 | 精确 | 精确 |
| 延迟/丢包模拟 | **不支持** | **支持**（netem） |
| 真实度 | **更真实** | 较真实 |
| 适用场景 | 正式实验、演示 | 开发调试、需模拟延迟时 |

> 两种方案可组合使用：交换机管带宽上限，tc 叠加延迟/丢包。**但不要在两处同时限带宽，否则会叠加。**

### 3.5 验证限速效果

```bash
# 1. iperf3 测带宽
iperf3 -s                          # 目标节点启动服务端
iperf3 -c <目标IP> -t 10           # 测试节点发起测试

# 预期结果：
#   边→云: ~64 Mbps  (8 MB/s)
#   端→边: ~800 Mbps (100 MB/s)

# 2. 实际文件传输验证
scp testfile.bin user@<云IP>:/tmp/  # 观察速率应 ~8 MB/s
scp testfile.bin user@<边IP>:/tmp/  # 观察速率应 ~100 MB/s
```

### 3.6 项目代码适配

使用物理限速（交换机或 tc）后，项目代码中的 `simulate_bandwidth_mbps` 应设为 `null`，让真实网络延迟自然生效：

```json
{
    "simulate_bandwidth_mbps": null
}
```

> **切勿同时使用物理限速和代码中的 simulate_transfer()，否则延迟会叠加。**

## 四、10GB 数据推理评估

### 4.1 数据规模

| 数据集 | 单样本大小 | 10GB 对应样本数 |
|--------|----------|---------------|
| Link11 (1024点, complex64) | 8 KB | **~125 万条** |
| RML2016 (600点, complex64) | 4.8 KB | **~208 万条** |
| Radar (500点, complex64) | 4 KB | **~250 万条** |

### 4.2 硬件能力评估

| 资源 | 需求 | RTX 4090 / 4070 Ti 是否满足 |
|------|------|---------------------------|
| GPU 显存 | 分批推理，每批 ~1MB | 完全满足（batch=128 逐批处理） |
| 系统内存 | 加载 10GB 数据 + 开销 ≈ 20-25GB | 需要 **32GB RAM** |
| 存储 | 10GB PKL 文件 | 需 SSD，HDD 读取会很慢 |

### 4.3 各环节耗时预估（Link11 10GB，基于实测数据线性缩放）

> 实测数据（5000样本）：纯云推理138.15s，协同推理6.11s（边5.77s+云0.34s），推理加速比22.6倍，卸载率0.25%。缩放系数k≈18.7。

**协同计算 T1：**

| 环节 | 计算方式 | 耗时 |
|------|---------|------|
| **端→边传输** | 10GB ÷ 100MB/s | **~102s** |
| **边侧推理** (2×RTX 4070 Ti) | 5.77s × 18.7 | **~108s** |
| **边→云传输** (0.25%低置信度 ≈ 25.6MB) | 25.6MB ÷ 8MB/s | **~3.2s** |
| **云侧推理** (RTX 4090, 0.25%样本) | 0.34s × 18.7 | **~6.4s** |
| **资源调度** (北邮) | 毫秒级 | **≤0.005s** |
| **T1 合计** | | **~219s ≈ 3.7 分钟** |

**纯云端计算 T2：**

| 环节 | 计算方式 | 耗时 |
|------|---------|------|
| **端→云传输** | 10GB ÷ 8MB/s | **~1,280s** |
| **云侧推理** (RTX 4090) | 138.15s × 18.7 | **~2,583s** |
| **T2 合计** | | **~3,863s ≈ 64 分钟** |

**T2/T1 = 3,863 / 219 ≈ 17.6 ✓✓✓，远超10倍要求。**

### 4.4 各环节时间占比（协同模式）

```
端→边传输 (102s)  ██████████████░░░░░░░░░░░░░░░░  47%
边侧推理 (108s)   ███████████████░░░░░░░░░░░░░░░  49%
边→云传输 (3.2s)  ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░   1%
云侧推理 (6.4s)   ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░   3%
调度 (<1s)        ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░  <1%
```

### 4.5 注意事项

1. **内存**：当前代码 `pickle.load()` 一次性加载全量数据到 RAM，10GB 数据需要 **32GB 内存**
2. **存储**：10GB PKL 文件建议放 SSD，HDD 读取 10GB 约需 1-2 分钟额外开销
3. **分批加载**：若内存不足，可改代码为分块加载（需修改 `io_helper.py`），需要时可实现

## 五、评估指标体系

### 5.1 指标公式

```
T_total = T_transfer + T_inference + T_scheduling

其中：
  T_transfer   = T_端→边 + T_边→云            （网络传输时间）
  T_inference  = T_边侧推理 + T_云侧推理        （计算推理时间）
  T_scheduling = T_任务分派 + T_资源分配         （资源调度时间，北邮负责，毫秒级）
```

### 5.2 关键结论

- 10GB 数据协同推理约 **3.7 分钟**，端→云直传约 **64 分钟**，**T2/T1 ≈ 17.6，远超10倍**
- 实测推理加速比达 22.6 倍，云端卸载率仅 0.25%，推理优势是达标的核心
- 协同模式中端→边传输（47%）和边侧推理（49%）占主导，散射链路传输占比极小（1%）
- 北邮毫秒级调度开销占比 < 0.01%
