## 二、测试项 TC-04-01：雷达辐射源识别数据处理任务

| 要素 | 内容 |
|------|------|
| **测试项编号** | TC-04-01 |
| **测试项名称** | 雷达辐射源识别数据处理任务协同计算执行时间验证 |
| **测试对象及考核指标** | 测试对象：云边端松耦合动态协同原型系统。考核指标：相比现有单独云端计算方式，云边端协同计算方式的执行时间减少一个数量级，即T2/T1≥10。 |
| **测试设备** | 云节点服务器1台（RTX 4090 24GB GPU，32GB内存）；边缘节点服务器2台（RTX 4070 Ti 12GB GPU，32GB内存）；端节点计算机2台（无GPU，8GB内存）；企业可网管千兆交换机1台；千兆网线7根。 |
| **测试方法** | 搭建由2个端节点（模拟小型无人机/船）、2个边缘节点（模拟大型无人机/船）、1个云节点组成的测试环境。通过交换机端口限速或Linux tc命令设置通信链路带宽：端节点与边缘节点之间的无线自组网链路限速为800Mbps（100MB/s），边缘节点与云节点之间、端节点与云节点之间的散射通信链路限速为64Mbps（8MB/s）。云节点部署基于复数卷积神经网络的大模型（complex_resnet50_radar_with_attention），边缘节点部署经知识蒸馏训练的轻量模型（real_resnet20_radar_h）。在同等计算和网络资源条件下，分别执行以下两种计算方式并记录执行时间：（1）**单独云端计算方式**——端节点将全量雷达辐射源数据通过散射链路直接传输至云节点，由云节点使用大模型完成全部识别任务；（2）**云边端协同计算方式**——端节点将数据通过自组网链路传输至边缘节点，边缘节点使用轻量模型进行推理并筛选低置信度样本，仅将低置信度样本通过散射链路转发至云节点，由云节点使用大模型进行二次推理，最终合并边侧与云侧识别结果。云边端协同执行时间记为T1，单独云端执行时间记为T2，若T2/T1≥10，则满足要求。 |
| **测试数据** | 雷达辐射源识别数据集（Radar，7类雷达辐射源信号），数据量不少于10GB，信号长度500点，complex64格式。 |
| **测试要求** | 1. 测试前使用iperf3工具验证各链路实际带宽符合设定值（端↔边约800Mbps，边↔云/端↔云约64Mbps）；2. 两种计算方式使用完全相同的测试数据集、相同的硬件资源和网络资源条件；3. 执行时间需完整记录三个分项：任务规划与资源调度方案生成耗时、数据传输耗时、任务计算耗时，其中执行时间 = 任务规划与资源调度方案生成耗时 + 数据传输耗时 + 任务计算耗时；4. 每种计算方式至少执行3次，取平均值作为最终结果。 |
| **合格判据** | 单独云端计算执行时间T2与云边端协同计算执行时间T1之比 T2/T1 ≥ 10。 |
| **测试流程** | 见下方详细流程。 |

### 测试流程

**准备阶段：**

1. 部署云边端松耦合动态协同原型系统至各节点，确认软件版本一致、各节点间网络连通
2. 配置通信链路带宽限速：
   - 方式一（交换机端口限速）：云节点所连端口入出限速64Mbps，边缘节点所连端口入出限速800Mbps，端节点所连端口不限速
   - 方式二（Linux tc命令）：在各节点分别执行 `sudo ./scripts/network_shaping.sh <角色>`，按目标IP区分链路速率
3. 使用iperf3在端→边、边→云、端→云三条链路上分别进行带宽测试，确认实测值与设定值一致
4. 将不少于10GB的雷达辐射源识别测试数据（PKL格式）均分到2个端节点本地存储
5. 确认云侧大模型（complex_resnet50_radar_with_attention）和边侧轻量模型（real_resnet20_radar_h）均已完成训练，权重文件就绪

**执行阶段——单独云端计算方式（记录T2）：**

6. 在云节点上启动推理服务，加载大模型
7. 通过资源调度模块生成任务规划方案，记录调度耗时T2_调度
8. 2个端节点同时通过64Mbps散射链路向云节点传输全量数据，记录传输耗时T2_传输
9. 云节点接收全部数据后，使用大模型对全量样本执行雷达辐射源识别推理，记录计算耗时T2_计算
10. 汇总 T2 = T2_调度 + T2_传输 + T2_计算

**执行阶段——云边端协同计算方式（记录T1）：**

11. 在云节点和2个边缘节点上分别启动推理服务，云节点加载大模型，边缘节点加载轻量模型
12. 通过资源调度模块生成协同任务规划与资源分配方案，记录调度耗时T1_调度
13. 2个端节点同时通过800Mbps自组网链路向各自对应的边缘节点传输数据，记录端→边传输耗时
14. 边缘节点使用轻量模型对接收数据执行推理，根据置信度阈值筛选出低置信度样本
15. 边缘节点将低置信度样本通过64Mbps散射链路发送至云节点，记录边→云传输耗时
16. 云节点使用大模型对接收到的低置信度样本执行二次推理
17. 合并边侧高置信度识别结果与云侧二次推理结果，得到最终识别结果
18. 汇总传输耗时 T1_传输 = 端→边传输耗时 + 边→云传输耗时，计算耗时 T1_计算 = 边侧推理耗时 + 云侧推理耗时
19. 汇总 T1 = T1_调度 + T1_传输 + T1_计算

**验证阶段：**

20. 计算 T2/T1，判断是否 ≥ 10
21. 重复步骤6-20共3次，取3次T2/T1的平均值
22. 记录测试结果，截屏保存各节点系统输出日志及timing文件

## 五、测试环境配置

### 5.1 网络拓扑与带宽设定

| 链路 | 通信方式 | 带宽 |
|------|---------|------|
| 端↔边 | 无线自组网 | 800 Mbps（100 MB/s） |
| 边↔云 / 端↔云 | 散射通信 | 64 Mbps（8 MB/s） |

### 5.2 硬件设备

| 设备 | 规格 | 数量 | 用途 |
|------|------|------|------|
| 云节点服务器 | i7-13700, 32GB RAM, RTX 4090 24GB, 1TB SSD | 1台 | 大模型推理 |
| 边缘节点服务器 | i5-13400, 32GB RAM, RTX 4070 Ti 12GB, 512GB SSD | 2台 | 轻量模型推理、低置信度筛选 |
| 端节点计算机 | i3-12100, 8GB RAM, 无GPU, 128GB SSD | 2台 | 数据采集/加载 |
| 企业可网管千兆交换机 | TP-Link TL-SG2008 或同级，8口千兆 | 1台 | 端口限速 |
| 千兆网线 | CAT6 | 7根 | 连接各节点到交换机 |

### 5.3 网络限速配置

**方式一：交换机端口限速**

| 交换机端口 | 入口限速 | 出口限速 | 连接设备 |
|-----------|---------|---------|---------|
| Port 1 | 64 Mbps | 64 Mbps | 云节点 |
| Port 2 | 800 Mbps | 800 Mbps | 边缘节点 1 |
| Port 3 | 800 Mbps | 800 Mbps | 边缘节点 2 |
| Port 4-5 | 不限 | 不限 | 端节点 1-2 |

**方式二：Linux tc 命令限速**

```bash
sudo ./scripts/network_shaping.sh cloud      # 云节点
sudo ./scripts/network_shaping.sh edge       # 边缘节点
sudo ./scripts/network_shaping.sh device     # 端节点
sudo ./scripts/network_shaping.sh clear      # 清除限速
```

### 5.4 软件环境

| 软件 | 版本 | 用途 |
|------|------|------|
| Ubuntu | 22.04 LTS | 操作系统 |
| Python | 3.10+ | 运行环境 |
| PyTorch | 2.5.1+cu121 | 深度学习框架 |
| CUDA | 12.1 | GPU 加速 |
| iperf3 | 3.x | 带宽验证 |

### 5.5 带宽验证方法

```bash
# 目标节点启动服务端
iperf3 -s

# 测试节点发起测试
iperf3 -c <目标IP> -t 10
```
